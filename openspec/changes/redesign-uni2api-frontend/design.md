## Context

Uni2API 当前为 Next.js 16 / React 19 / Tailwind CSS 4 前端和 FastAPI 后端。根布局使用横向 `TopNav`，根路由按角色跳转至 `/accounts` 或 `/image`；主题存储和本地会话仍使用旧品牌 key。账户、图片、日志、代理运行时、设置和图像任务已拥有独立 API，但没有能直接支撑管理员工作台的最小聚合模型。

本次变更横跨应用壳层、导航、主题、首页数据、创作流和现有管理页。视觉和产品决策已固定在 `docs/design/uni2api-frontend-redesign.md` 与 `docs/design/assets/uni2api-admin-dashboard-soft-ui.png`，实现必须以其为验收基线。

约束：继续使用 pnpm、shadcn/ui、21st.dev Community Components、Uiverse Elements、Radix、Lucide、Zustand、localforage 和 Sonner；保留现有 URL、认证角色和后端业务 API 语义；不为装饰性 UI 引入大型依赖。

## Goals / Non-Goals

**Goals:**

- 将现有页面统一到角色感知的 Soft UI 应用壳层，并保留所有当前路由。
- 让管理员在首页获取可信、最小化的运行状态与异常处理入口；让普通用户优先进入创作与个人任务。
- 复用现有图像任务、会话、账户、图片和设置能力，避免重写领域逻辑。
- 建立可复用的主题 tokens、组件状态、响应式和无障碍规则。
- 建立可审计的三来源组件选择和本地适配机制，使 shadcn/ui、21st.dev 和 Uiverse Elements 在界面中呈现为同一个 Uni2API 组件系统。
- 用有界、管理员授权的汇总接口提供全局运行信号，避免浏览器拼接海量日志。
- 在每条现有路由完成替换验证后删除旧前端实现、遗留视觉规则和旧品牌存储，保持代码库只有一套活跃呈现层。

**Non-Goals:**

- 不修改 OpenAI 兼容 API、账号轮询、图像生成协议、存储后端或认证模型。
- 不实现自定义拖拽布局、实时推送通道、告警订阅或新的权限角色。
- 不在本变更内重写所有数据表的服务端分页；仅在已有客户端上实施必要的视觉和交互改造，并为后续虚拟化留出边界。
- 不将静态草图中的示例图片或第三方设计素材作为运行时产品资产。
- 不删除现有公开 URL、认证/图像/账户业务能力、上游仓库 URL、兼容性配置值或用户持久化数据；这些内容只在具备明确迁移时更改。

## Decisions

### 1. 使用应用壳层取代横向顶栏

新增 `ConsoleShell`，由根布局在认证后页面中承载。它负责品牌、侧栏、移动抽屉、全局工具区、命令面板、主题和用户退出；页面本身只拥有内容区和二级导航。

- 路由与权限映射在单一 `navigation` 配置中定义，管理员专属项在渲染和命令搜索两处同时过滤。
- `TopNav` 将被替换或拆分为该壳层的子组件；登录页不渲染应用壳层。
- 桌面 `≥1280px` 展开侧栏，`768–1279px` 收窄成有 tooltip 的图标轨，`<768px` 使用 Sheet 抽屉。

替代方案：保留横向导航并新增下拉更多菜单。拒绝原因：设置、调试和管理入口会继续争夺宽度，且无法为创作页留下稳定画布。

### 2. 用受角色保护的工作台汇总端点提供运行状态

新增 `GET /api/dashboard/summary`。端点通过 `require_identity` 识别角色；管理员返回账户健康、未完成任务数、近 24 小时调用统计、代理运行时摘要、最新图片和可行动的警告；普通用户只返回该身份拥有的未完成任务、最近任务和可用模型，不返回账户池、日志、代理或其他用户数据。

- 汇总在服务端通过 `account_service.account_health()`、`image_task_service`、`log_service`、`proxy_settings.get_runtime_status()` 和受限图片查询计算。
- 近 24 小时成功率以 `log_service` 中 `LOG_TYPE_CALL`、具备 `detail.status` 的记录为来源；没有可计数调用时返回 `null` 与明确空状态，而不是伪造 100%。
- 响应包含 `severity`、`code`、`title`、`detail`、`action`、`href`，前端只负责呈现和跳转，不解析错误文本决定阻断状态。
- 端点为只读、无缓存写入，前端在进入工作台时加载；有未完成任务或运行中账号刷新时以短周期轮询更新，其他时间由“刷新”动作或页面重访更新。

替代方案：前端并行拉取 `/api/accounts`、`/api/logs`、`/api/image-tasks` 和 `/api/proxy/runtime` 后计算。拒绝原因：普通用户权限边界复杂、日志最多 200 条不完整、数据时点不一致且扩大浏览器负担。

### 3. 首页快捷创作只创建草稿并交给现有创作页

新增 `workspace-quick-create` 本地草稿协议，内容包括提示词、模型、尺寸/画幅、质量、数量和参考图元数据。工作台保存草稿后导航到 `/image?draft=<id>`；`/image` 消费草稿、使用既有会话/任务编排和 `createImageGenerationTask` 创建任务，再清理已消费草稿。

- 草稿通过现有 localforage 存储实例或独立实例持久化，不能放进 URL 或服务端日志，以免泄露提示词和本地 Data URL。
- 初版快捷创作只提交单张文生图给现有异步任务服务；数量选择为进入完整页后的会话默认值，批量多图仍由现有页面流程处理。实现前需要让 `ImageTaskService.submit_generation` 接受经过验证的 `n`，或明确保留单图限制，不能在 UI 显示无法兑现的数量。
- 工作台在汇总端点指示阻断性异常时禁用创建并把主动作切换到诊断链接。

替代方案：首页直接复制 `/image` 的完整 composer 与轮询逻辑。拒绝原因：形成两套状态源，增加会话、滚动位置和失败重试不一致风险。

### 4. 以语义 token 实现平衡型 Soft UI，而非页面级覆盖

在 `globals.css` 中建立浅/深主题语义变量：画布、浮起表面、按入表面、文本、primary、creative、success、warning、danger、focus、阴影、圆角和动效。现有通用 Button、Input、Tabs、Sheet、Dialog、Card、Table 等组件先改为引用 token；页面使用语义 class 或轻量组合，而不是在每页散落原始 stone/brown 色值。

- 外凸双向阴影仅用于同色表面的壳层、主要面板、按钮和导航项；输入、选中导航和分段控制使用内阴影。
- 数据表格行、日志行、chip 和密集设置项保持平整，使用低对比分割与 hover 背景，不重复叠加阴影。
- 深色主题是单独映射，不复用浅色高光；主题存储键迁移为 `uni2api-theme`，读取旧 `chatgpt2api-theme` 一次后迁移以保留用户偏好。
- 所有普通文本前景组合需达到 WCAG AA `4.5:1`；聚焦状态使用可见 2px ring，不只依赖阴影。

替代方案：仅在页面 wrapper 覆盖颜色。拒绝原因：现有 UI 组件会继续输出棕灰边框、圆角和主题行为，跨页不一致。

### 5. 按页面优先级迁移，保持 API 与 URL 稳定

实施顺序为：基础 tokens/壳层 → 工作台/快捷创作 → 图像创作与图片库 → 账户池与日志 → 设置与实验室 → 登录页。每个页面先保留请求函数、权限 guard、客户端状态和 URL，再调整视图层；组件提取只围绕重复交互进行。

- 账户池和日志保留当前批量操作、筛选和详情；表格外层进入 Soft UI 容器，行保持紧凑。
- 设置从横向主标签改为页面内二级目录，但每个 `value` 与现有 store 初始化方式不变。
- 调试页移动到折叠“实验室”，路由 `/debug` 不变。
- 21st.dev 候选仅作为 Sidebar、Table、Upload、Tabs、Dialog、Empty state 的结构参考；实施继续用 shadcn/Radix，除非候选在依赖、可访问性和适配成本上明确胜出。

### 6. 用路由级验证和断点矩阵控制风险

前端每次迁移先运行 `pnpm` 类型检查/构建，后端汇总端点运行 `uv` 测试。浏览器验证覆盖 `375 / 768 / 1024 / 1440px`，至少涵盖管理员正常、管理员阻断、普通用户、空数据、加载、深色和 reduced motion 状态。

替代方案：仅靠单张桌面截图验收。拒绝原因：侧栏折叠、表格降级和移动抽屉是本变更的核心行为。

### 7. 以替换映射驱动旧前端退役

现有 `page.tsx`、`login/page.tsx`、`image/page.tsx`、`image-manager/page.tsx`、`accounts/page.tsx`、`logs/page.tsx`、`settings/page.tsx` 和 `debug/page.tsx` 均为保留 URL 的实现位置，不会因“清理页面”而删除路由文件。每个页面在迁移时必须拆出或重写旧视图结构，之后删除已经被新模块替代的 JSX、局部样式、状态和导入。

退役工作采用以下门槛：

1. 在 [旧前端路由退役映射](./legacy-route-retirement-map.md) 中建立旧实现到新模块的替换映射，标明路由、权限、数据源、交互和验证证据。
2. 新壳层上线并通过管理员、普通用户、移动端与深链验证后，删除 `TopNav`、`HeaderActions`、旧 `ThemeToggle`、旧 `ThemeScript` 和旧动画主题切换实现，或仅保留被新壳层直接复用且已改名/归属清晰的部分。
3. 通过全仓引用审计后删除 `text-3d-flip.tsx`、`morphing-text.tsx`、`dia-text-reveal.tsx` 等无人引用的视觉组件；若实施期间新增真实使用者，则保留并将其纳入新动效规范。
4. 新 token 覆盖所有路由并完成视觉回归后，删除旧棕色径向背景、`.dark` 下针对 `bg-white`/`stone-*` 的强制覆盖、未使用的 `!important` 补丁和已废弃页面 wrapper 样式；不得仅以全局替换删除仍被未迁移页面使用的规则。
5. 主题、认证和图像会话等旧命名空间存储必须先读旧值、原子写入新命名空间、确认成功后才删除旧值。存储迁移失败时继续使用旧值且不能丢失用户会话或草稿。

品牌清理只针对用户可见前端名称和前端存储命名空间。`basketikun/chatgpt2api` 上游链接、`chatgpt2api/images` WebDAV 默认路径、兼容 API 文档示例及调试 Skill 名称均是兼容性边界，除非另有批准的迁移方案，不纳入本次删除。

替代方案：重设计完成后再凭搜索结果手工删除。拒绝原因：容易误删兼容性值或保留死组件，且没有可审计的退役时点。

### 8. 用三来源组件治理保持界面统一

每个新增或重构的前端组件 MUST 先建立组件来源记录，再进入代码实现。来源记录至少包含目标组件、功能和状态、候选链接/作者/许可证、功能匹配、栈匹配、视觉匹配、响应式、无障碍、适配成本、维护风险、接受/拒绝理由和最终本地组件路径。记录保存在本变更的 [组件来源清单](./component-sourcing-register.md) 中，并在实施完成后归档到设计文档。

来源职责固定如下：

| 来源 | 首选范围 | 不可替代的责任 | 采用方式 |
| --- | --- | --- | --- |
| shadcn/ui | Button、Input、Textarea、Select、Checkbox、Dialog、Sheet、Popover、Tabs、Table、Calendar、Toast 等语义基础原语 | Radix 行为、键盘交互、焦点管理与 ARIA 基线 | 保持现有 API，改造其 class/tokens；不能用 Uiverse 或 21st 原始 DOM 替换这些行为基础 |
| 21st.dev Community Components | Sidebar、命令面板、Dashboard 区块、空状态、上传/预览、数据表外壳、设置导航等复合结构 | 按功能、交互、视觉三个维度先搜 2–4 个候选 | 选中后将结构和交互本地化，换成 shadcn 原语和 Uni2API tokens，删除演示文案、无关依赖与装饰 |
| Uiverse Elements | Soft UI 按钮表面、输入按入效果、开关/checkbox 反馈、loader、tooltip、卡片微交互等视觉或微交互候选 | 为平衡型 Soft UI 提供经筛选的可触摸细节 | 仅复制经允许的本地 CSS/最小标记；保留或记录作者/许可证；不能引入远程 CSS/JS、不可访问 click-div、连续装饰动画或破坏布局的效果 |

实现边界：

- `web/src/components/ui/` 是唯一的基础原语出口；其 props、语义元素、Radix 交互与可访问性约定为稳定 API。
- `web/src/components/console/` 承载由 21st/Uiverse 结构改造而来的业务复合组件，例如 `ConsoleSidebar`、`DashboardSignalGroup`、`QuickCreatePanel`、`CommandMenu`、`SoftSurface`、`EmptyState` 和 `UploadPreview`。
- 任何外部候选均必须移除原始 palette、字体、圆角、阴影、动画时间、图标和 demo 数据，并改用 `--ui-*` 语义 tokens、Lucide 和已有数据流。
- 禁止运行时从 21st.dev 或 Uiverse 加载 CSS/JS；禁止直接复制外部的 `onClick` 容器替换 button/input/dialog 等语义原语；禁止引入与现有 Radix 或 motion 依赖重叠的组件库。
- Uiverse 的当前访问若受 Cloudflare 或其他来源限制，实施者 MUST 记录“来源暂不可访问”，不得伪造候选；应在可访问会话中完成选择，或采用已审查的本地 shadcn/21st 候选并将 Uiverse 项标记为待补选。

候选选择流程：

1. 按功能、交互和 Soft UI 风格分别搜索 21st.dev 与 Uiverse；每类至少保留 2 个可比较候选，除非来源不可访问并记录原因。
2. 使用功能适配、技术栈适配、视觉适配、响应式、无障碍、适配成本和维护风险七项矩阵评分；不因展示效果优先选择高维护候选。
3. 确认最终来源后，本地适配到 shadcn 基础原语和 Uni2API tokens，并补齐 default、hover、focus-visible、active、disabled、loading、error 状态。
4. 在组件展示页或组件测试中以浅色、深色、键盘、减少动效和小屏状态验证；未达到 4.5:1、44px 触摸目标或可见焦点的候选必须拒绝或修复。

初始清单固定覆盖 `ConsoleShell`、`ConsoleSidebar`、`CommandMenu`、`DashboardSignalGroup`、`QuickCreatePanel`、`UploadPreview`、`SoftSurface`、`EmptyState`、`DataTableShell` 和 `SettingsNav`。其中的 21st.dev 短名单仅表示已发现的候选，不等于已经采用；Uiverse 在来源可访问、作者和许可证核验完成前保持待补选状态。任何实现任务开始前，必须将清单中对应组件的裁决更新为可审计结论。

替代方案：直接在不同页面混用三个库的原始组件。拒绝原因：会产生三套 palette、阴影、圆角、动效和可访问性行为，违反本次“统一组件”的目标。

## Risks / Trade-offs

- [汇总端点泄露管理员运行信息] → 使用 `require_identity`，按角色构建响应；为普通用户响应编写字段级测试，禁止复用管理员 DTO。
- [日志统计扫描增大请求延迟] → 限定 24 小时窗口并在 `LogService` 添加专用聚合方法，避免向前端传输原始日志；数据规模增长后可迁移到存储层聚合。
- [Soft UI 降低密集管理页可读性] → 阴影仅用于容器；表格和日志行使用平整样式、清晰分割和已验证对比度。
- [旧品牌本地存储键造成偏好丢失] → 为主题、认证和图像会话设置迁移策略；未经迁移验证前不删除旧键读取。
- [首页与创作页形成双重任务逻辑] → 首页只存草稿和跳转；创建、轮询、重试和会话保存只在 `/image` 执行。
- [逐页重设计导致长期分支难合并] → 以壳层和 tokens 先落地，每页独立可验证；保持路由、API 和 store 合同不变。
- [移动端复杂管理体验退化] → 明确支持搜索、筛选、详情和单项操作；对批量导入、跨列比较和高级配置使用有解释的桌面提示，而不是隐藏功能。
- [清理旧样式时影响尚未迁移页面] → 以替换映射为门槛，按路由完成迁移和截图验证后再删除相应规则；每次删除运行构建和路由 smoke test。
- [重命名本地存储导致用户会话或草稿丢失] → 先迁移再删除旧键，迁移失败保留旧命名空间并提供回归测试。
- [全局替换历史名称破坏兼容配置] → 将可见品牌/本地存储与上游 URL、WebDAV 路径、API 示例和 Skill 标识分为不同清单，后者默认保留。
- [三方组件视觉和行为不一致] → 以 shadcn 原语 API 与 Uni2API token 为唯一出口，所有外部候选先本地适配并通过组件来源清单审查。
- [Uiverse 候选缺少许可证或无障碍基础] → 记录作者/链接/许可证；仅采用可本地化、可审计的 CSS/标记，并以 shadcn 语义原语包裹，缺失信息或无法修复则拒绝。
- [来源站点不可访问导致虚构组件选择] → 将来源不可用记录为阻塞事实，延后该来源选择或使用已通过审查的替代候选，绝不编造组件链接或能力。

## Migration Plan

1. 在独立 feature branch 上完成 tokens、壳层和工作台汇总端点，并保留当前导航作为短期回退实现直到路由验收通过。
2. 将根路由默认目标改为 `/` 工作台，同时保留角色 guard 和所有现有深链。
3. 分页迁移已有页面；每个页面通过构建、路由访问、角色权限和桌面/移动截图检查后再迁移下一页。
4. 为基础原语、应用壳层和每个页面复合组件完成三来源候选选择、本地适配与组件展示验证；拒绝或暂缓的候选必须记录原因。
5. 对每条已迁移路由执行替换映射审计，移除其旧 JSX/CSS/组件引用；所有路由迁移后删除旧导航、主题和无人引用动画组件。
6. 主题、认证和图像会话键采用读旧写新的方式迁移；确认写入成功后删除旧键并验证会话、主题和草稿持续存在。
7. 发布前以真实或脱敏的账户、任务、日志数据验证正常、降级、阻断和空状态，并运行前端残留审计。

回滚：若应用壳层或工作台接口导致生产问题，恢复旧根布局/`TopNav` 和原角色默认路由；汇总端点是新增只读接口，可独立保留或移除，不影响既有图像和账户操作 API。
